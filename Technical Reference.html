Technical Reference

gemini-2.0-flash-live-001

A comprehensive developer's guide for integrating and maintaining applications using the `gemini-2.0-flash-live-001` model.

Core Model Specifications

This model is a stable, earlier version within the Gemini family, specifically designed for real-time streaming applications. Its key characteristic is its architecture, which dictates its capabilities and ideal use cases.

Architecture

Half-Cascade Audio

A modular pipeline with native audio input and a separate Text-to-Speech (TTS) model for output.

Primary Advantage

Production Reliability

Offers higher stability, especially for Tool Use and Grounding with Google Search.

Ideal Use Case

Enterprise & Legacy Systems

Best for mission-critical agents or existing systems where this version is integrated.

Known Limitation

No Advanced Features

Lacks access to Affective Dialog, Proactive Audio, and other Native Audio features.

Data Handling & Modalities

The model has strict requirements for data formats to ensure low-latency streaming. Adhering to these specifications is critical.

Supported Modalities (Live API)

  • ✓Input: Streaming Audio, Text
  • ✓Output: Streaming Audio, Text
  • ✗Not Supported: Direct Video/Image input.

Session & Context Limits

  • Audio-only Session: ~15 min limit.
  • WebSocket Connection: ~10 min limit.
  • Long-running Sessions: Must enable Context Compression and Session Resumption.
  • Disconnection Notice: Server sends a `GoAway` message.

Audio Format Specifications

StreamSample RateEncodingChannels
Input (Client → Server)16000 Hz16-bit PCM (little-endian)1 (Mono)
Output (Server → Client)24000 Hz16-bit PCM (little-endian)1 (Mono)

Design Rationale: Asymmetric Sample Rates

The 16kHz input is optimized for bandwidth efficiency. The higher 24kHz output allows the model to generate more crisp, high-fidelity audio, optimizing for user experience.

Voice & Speech Control

This model supports a specific, curated set of 8 voices. Using a voice name not on this list will result in an error.

Control Paradigm: Declarative vs. Expressive

This model's Half-Cascade architecture uses a **declarative** control paradigm (selecting a voice name). This contrasts with Native Audio models, which use an **expressive** paradigm (e.g., "speak more cheerfully").

Important: The voice 'Sadaltager' is not compatible with this model. You must use one of the voices below.

API Usage & Interaction Dynamics

Interaction occurs over a stateful WebSocket connection. Understanding the interruption protocol is key.

Interruption Protocol

  • User interruption immediately cancels the model's in-flight generation.
  • The server sends a `BidiGenerateContentServerContent` message to report the interruption.
  • In-progress tool calls are also canceled and reported.

Request Structure Example

The following shows how to configure a session to use a specific voice.


# Python SDK Example
import google.genai as genai
from google.genai import types

# Must select a voice from the compatible list
voice_name = "Puck" 

config = types.LiveConnectConfig(
    speech_config=types.SpeechConfig(
        voice_config=types.VoiceConfig(
            prebuilt_voice_config=types.PrebuiltVoiceConfig(
                voice_name=voice_name,
            )
        ),
    ),
    # Other configs like VAD settings go here
)

# Use this config when connecting
# client.aio.live.connect(
#   model='models/gemini-2.0-flash-live-001',
#   config=config
# )
                                

// Equivalent JSON configuration payload
{
  "speechConfig": {
    "voiceConfig": {
      "prebuiltVoiceConfig": {
        "voiceName": "Puck"
      }
    }
  },
  "realtimeInputConfig": {
    "automaticActivityDetection": {
      "startOfSpeechSensitivity": "START_SENSITIVITY_LOW",
      "endOfSpeechSensitivity": "END_SENSITIVITY_LOW"
    }
  }
}

Authentication & Quotas

  • Authentication: Standard Google Cloud auth. A server-to-server implementation is recommended.
  • Quotas: Subject to standard Vertex AI API quotas. Check your GCP project for limits.

Versioning & Future-Proofing

While stable, this is an older model. Consider the following for long-term maintenance.

Migration to Newer Models

  • To `gemini-live-2.5-flash-preview`: Recommended path for staying on the Half-Cascade architecture. Largely compatible API.
  • To Native Audio (`gemini-2.5...-native-audio...`): More significant migration. Unlocks a larger voice library and prompt-based style control.

Best Practices for Code Updates

  • Isolate Model Name: Store the model name in a configuration variable, not hardcoded.
  • Abstract Voice Selection: Create a helper function for voice configuration to simplify future migrations.
  • Monitor Documentation: Regularly check official Google Cloud docs for deprecation notices.

Implementation Pathway

Recommended path: **High-Reliability Enterprise Agent** where stability is paramount.

  • Architecture: Use the native **Half-Cascade** architecture for its documented stability.
  • Session Management: Implement **Context Window Compression** and **Session Resumption** from the start.
  • VAD Tuning: Configure Voice Activity Detection for lower sensitivity to prevent accidental interruptions.
  • Implementation: A **Server-to-Server** connection is strongly recommended to secure credentials.
